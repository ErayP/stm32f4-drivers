# STM32F407 Bare-Metal Drivers

This repository contains my bare-metal driver development work for the STM32F407 microcontroller.

The purpose of this project is to improve my understanding of low-level embedded software development by writing peripheral drivers using register-level programming in C.

## Target Platform

- **MCU:** STM32F407VG
- **Architecture:** ARM Cortex-M4
- **Programming Language:** C
- **Development Style:** Bare-metal / register-level programming

## Implemented Drivers

- GPIO
- SPI
- I2C

## Project Structure

- `drivers/Inc/` -> driver header files
- `drivers/Src/` -> driver source files
- `Src/` -> application/test files
- `Startup/` -> startup code
- `Debug/` -> build output files generated by the IDE/toolchain

## I2C Driver

The I2C driver includes basic peripheral initialization and communication support, with additional interrupt-based receive handling.

### Features
- I2C peripheral initialization
- Basic master communication support
- Interrupt-based master receive handling
- ACK management during reception
- Buffer length tracking during interrupt-driven transfers
- Peripheral state management during communication

## I2C Interrupt-Based Receive Details

One of the main focuses of this project is understanding how interrupt-driven I2C communication works on STM32 at the register level.

In the I2C receive flow, the driver handles communication step by step inside the interrupt/event handling process.

### Implemented Concepts
- Starting interrupt-based master reception
- Tracking the receive buffer and remaining data length
- Handling I2C event flow during reception
- Managing ACK behavior depending on the number of bytes to receive
- Generating STOP condition when reception is completed
- Resetting the peripheral state after transfer completion

### Notes
- The interrupt-driven receive logic is implemented to better understand low-level peripheral behavior rather than relying only on high-level abstraction libraries.
- Special care is required in I2C reception when receiving the last bytes, especially for ACK control and STOP generation.
- The driver keeps track of the communication state to prevent conflicts between ongoing operations.
